เอกสารนี้เป็นบันทึกแนวทางสำหรับขัดเกลาหนังสือภายหลัง เป้าหมายคือทำให้หนังสือไม่ใช่แค่มีโค้ดครบ แต่ทำให้ผู้เรียนเข้าใจว่ากำลังเขียนอะไร ทำไมต้องเขียน และตรวจได้อย่างไรว่าถูก

## หลักการสำคัญ

หนังสือสอนเขียนเว็บที่ดีควรทำให้ผู้เรียนสร้างของที่รันได้จริงเร็ว แต่ไม่ปล่อยให้พิมพ์ตามแบบไม่เข้าใจ

ทุกครั้งก่อนใช้ concept ใหม่ เช่น Controller, Service, Repository, DTO, Validation, Authentication, Docker หรือ Deployment ควรอธิบายก่อนว่า:

- มันแก้ปัญหาอะไร
- ถ้าไม่มีมันจะเกิดปัญหาอะไร
- ในงานจริงใช้มันเพื่ออะไร
- ตอนนี้เราจะใช้เฉพาะส่วนไหนก่อน

## โครงสร้างที่ควรมีในทุกบท

แต่ละบทควรมี:

- เป้าหมายของบทนี้
- ก่อนเริ่มต้องมีไฟล์หรือความรู้ใด
- สิ่งที่จะใช้ในบทนี้ พร้อมคำอธิบายสั้น ๆ
- ขั้นตอนสร้าง folder/file ที่ copy ได้
- โค้ดที่เพิ่มทีละส่วน ไม่ใช่แปะยาวรวดเดียว
- คำอธิบาย method, attribute, annotation หรือ configuration ก่อนใช้งาน
- คำสั่ง run/test/build ที่ใช้ตรวจ
- response หรือผลลัพธ์ที่ควรเห็น
- checkpoint ก่อนอ่านบทต่อไป

## วิธีสอนโค้ด

โค้ดควรถูกสอนเป็นชั้น:

1. สร้าง folder
2. สร้าง file
3. เพิ่ม using/import
4. เพิ่ม class/record/interface
5. เพิ่ม property
6. เพิ่ม method ทีละ method
7. ต่อเข้ากับ dependency injection หรือ configuration
8. ทดสอบผลลัพธ์

ถ้าโค้ดยาวเกินไป ให้แตกเป็นหลาย block และอธิบายระหว่างทาง

## สิ่งที่ต้องระวัง

- อย่าให้นักเรียนพิมพ์ตามโดยไม่รู้ว่า attribute หรือ method คืออะไร
- อย่าข้ามขั้นสร้าง folder/file
- อย่าใช้ port ตัวอย่างที่ไม่ตรงกับ project จริง
- อย่าใช้คำว่า production-ready ถ้ายังมี secret ใน Git หรือ test/build ไม่ผ่าน
- อย่าทำให้ demo ดูเหมือน production โดยไม่บอกข้อจำกัด

## Project ตัวอย่าง

หนังสือควรมี project ตัวอย่างอย่างน้อยสองแบบ:

- progressive หรือ validation project: ทำตามหนังสือทีละบท
- final project: โค้ดสุดท้ายที่สะอาดกว่าและใช้เป็น reference

ทั้งสองแบบควร build/test ผ่านจริง

## Mindset ที่ต้องสอน

หนังสือควรสอนเหตุผล ไม่ใช่แค่ syntax:

- ทำไมต้องแยก DTO
- ทำไมไม่ควรคืน entity ตรง ๆ
- ทำไม validation อยู่ที่ขอบระบบ
- ทำไม secret ห้ามอยู่ใน Git
- ทำไม token ต้องมีอายุสั้น
- ทำไม refresh token ต้อง rotate
- ทำไมต้องมี audit log
- ทำไม Docker image ไม่ควรรันด้วย root user

## นิยามคุณภาพ

หนังสือที่ดีควรทำให้ผู้เรียน:

- อ่านแล้วรู้ว่าต้องทำอะไรต่อ
- copy คำสั่งแล้วรันได้จริง
- เข้าใจ error ที่พบบ่อย
- ตรวจงานตัวเองได้จาก checkpoint
- เห็นความต่างระหว่าง demo, portfolio-ready และ production-grade

## เป้าหมายสุดท้าย

ผู้เรียนควรรู้สึกเหมือนมีครูที่นั่งข้าง ๆ:

- บอกลำดับให้ชัด
- อธิบายคำใหม่ก่อนใช้
- ไม่โยนโค้ดยาวเกินไป
- บอกวิธีตรวจว่าถูก
- บอกข้อจำกัดอย่างตรงไปตรงมา
